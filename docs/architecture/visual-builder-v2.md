# Visual Builder Architecture (v2)

> **Source of Truth**: [SYSTEM_MODEL.md](../../SYSTEM_MODEL.md) Section 3
> **Reference**: [VISUAL_BUILDER.md](../modules/VISUAL_BUILDER.md) (Legacy Architecture)

## Purpose

This document outlines the architecture for the next-generation AWCMS Visual Builder (v2).
It introduces an "Elementor-like" CSS-first paradigm, moving away from rigid layout preset blocks (`Section`, `Grid`)
toward a fluid, recursive `Container` model using Puck's `<Slot>` capabilities and dynamic inline styling.

## Audience

* Admin developers extending Puck editor configurations.
* Public portal developers creating custom Astro renderers (`PuckRenderer`).

## Core Concepts

1. **Atomic Container Model**: All layouts (columns, grids, wrappers) are built using a single, recursive `Container` block.
2. **CSS-First Configuration**: Layouts are determined by configuring CSS Flexbox and Grid properties directly on the Container, not by choosing a semantic block type.
3. **Standardized Styling (HOC)**: Every block (Text, Image, Container) exposes a standardized set of styling fields for margin, padding, typography, and borders via a Higher-Order Component/Factory pattern.
4. **Responsive Breakpoints**: Styling values can be defined per viewport (Desktop, Tablet, Mobile), compiling into reactive Tailwind utilities or specific CSS.

## 1. The Container Model

The legacy system relied heavily on hardcoded layouts (e.g., a `SectionBlock` with `paddingTop` and a `containerWidth` wrapper, or a `GridBlock` forcing a fixed column count).

The new `Container` block serves as the universal layout primitive.

### Recursive `<Slot>` Design

The `Container` block primarily leverages Puck's intrinsic `slot` field type.

```javascript
// AWCMS Admin: Container Block Schema
{
  fields: {
    content: {
      type: "slot",
      label: "Container Content"
    },
    // ... Layout & Style Fields (Display, Flex, Grid)
  },
  render: ({ content: Content, ...styleProps }) => {
    // Generates style object based on properties
    const inlineStyles = generateInlineStyles(styleProps);
    return (
      <div style={inlineStyles}>
        <Content />
      </div>
    );
  }
}
```

By allowing a `Container` to be dropped inside the `content` slot of another `Container`, authors can create infinitely complex UI trees (e.g., A Hero section containing a 2-column grid, where the left column is a flexbox column containing text and a button).

## 2. Standardized Styling Data Structure

To ensure consistency, advanced styling is decoupled from specific blocks.

### The Standardized Schema

Instead of every block defining its own padding fields, we inject a standard configuration object.

```javascript
const advancedStyleFields = {
  marginTop: { type: 'text', label: 'Margin Top' }, // e.g., '16px', '2rem', or responsive object (future)
  paddingBottom: { type: 'text', label: 'Padding Bottom' },
  backgroundColor: { type: 'text', label: 'Background Color' },
  // ... borders, typography
};

// Injecting into a standard block
const TextBlock = {
  fields: {
    text: { type: 'textarea' },
    ...advancedStyleFields
  }
}
```

## 3. The Renderer Pipeline (Astro)

The `awcms-public` portal is responsible for taking the massive `puck_layout_jsonb` payload and rendering it securely without loading the React editor runtime.

### The Layout Parsing Flow

1. **Fetch & Sanitize**: The `PuckRenderer.astro` component receives the JSON string from Supabase.
2. **Registry Match**: It recursively matches the JSON node type (e.g., `Container`, `Text`) to the local Astro component registry.
3. **Style Extraction**: `ContainerBlock.astro` receives the generic style props. It uses a utility function `parseStylesToCSS(props)` to map the JSON configuration into valid `style=""` strings or generated Tailwind classes.
4. **Slot Rendering**: `PuckRenderer` recursively calls itself to render the contents of any `slot` fields found inside a `Container`.

## 4. Multi-Tenancy & Security Adherence

The v2 architecture inherently respects the AWCMS rules:

* **Tenant Scoping**: All configurations generated by the `Container` block remain as plain JSON inside the `puck_layout_jsonb` column of the specific row edited by the tenant. No shared CSS files or global overrides are generated.
* **Editor Security (ABAC)**: Access to modify the layout (meaning access to the React `Puck` editor component) continues to be guarded by `awcms/src/components/dashboard/VisualPagesManager.jsx` using the `checkAccess(...)` policies.
* **Public Security (XSS)**: The Astro `PuckRenderer` continues to enforce a strict allow-list. Even if an attacker injects a `<script>` tag into a raw styling field, the strictly typed `parseStylesToCSS` utility will discard it as invalid CSS, maintaining XSS protection.

## 5. Migration Strategy

To ensure zero downtime or broken layouts for existing tenants:

1. **Phase 1: Coexistence**: `Container` is introduced alongside `Section` and `Grid`. Legacy blocks are hidden from the "Insert" menu (`disallow` config or custom category) but remain in the registry so existing layouts continue to render.
2. **Phase 2: Transition Tooling**: (Optional/Future) Admin script to traverse `puck_layout_jsonb` and convert `Section` nodes into `Container` nodes.
3. **Phase 3: Deprecation**: Once all tenants are transitioned, legacy blocks are removed from the codebase.
